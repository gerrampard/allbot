# 869 协议与框架收消息适配报告

> 本文档详细记录 869/8069 协议与 xbot 框架在 **从登录到等待接收消息** 的完整流程，以及 **WebSocket 收消息** 链路上的适配改动，包括消息格式、字段映射、门禁与入队逻辑，以及所有涉及的功能点。  
> 适配完成后：WS 能正确接收 869 推送、解析发送者与内容、入队并交由插件处理，**回复能发往真实发送者**。

---

## 0. 从登录到等待接收消息的完整流程（6 步）

框架在 `bot_core.py` 中按顺序执行以下 6 步，每步都有日志标记 `[流程-N]`，便于排查。

| 步骤 | 日志标记 | 含义 |
|------|-----------|------|
| 1 | `[流程-1]` | 配置与客户端就绪，进入登录阶段 |
| 2 | `[流程-2]` | 等待登录（扫码或恢复会话） |
| 3 | `[流程-3]` | 登录成功，进入初始化阶段 |
| 4 | `[流程-4]` | 初始化：心跳、机器人、数据库、插件 |
| 5 | `[流程-5]` | 就绪，即将进入消息接收 |
| 6 | `[流程-6]` | 等待消息（WebSocket 或 HTTP 轮询） |

### 步骤 1：配置与客户端就绪（流程-1）

**做什么**：加载配置、创建 869 客户端、设置管理后台 bot 实例，为登录做准备。

**代码位置**：`bot_core.py` 约 598–682 行。

**具体动作**：

1. **读配置**  
   - 从 `main_config.toml`（或 ConfigManager）读取 `WechatAPIServer`、`Protocol.version`、`XYBot` 等。  
   - 协议版本：`config.get("Protocol", {}).get("version", "ipad").lower()`，869 时为 `"869"`。

2. **创建 869 客户端**  
   - `create_wechat_client(..., protocol_version=protocol_version, ...)` 返回 Client869 实例（`bot`）。  
   - 869 使用 `WechatAPI.Client869`，鉴权用 `admin-key` / `device-key`。

3. **提前挂到管理后台**  
   - `set_bot_instance(bot)`，使登录前即可调用 get_qr_code 等接口。

4. **状态与日志**  
   - `update_bot_status("waiting_login", "等待微信登录")`。  
   - 打日志：`[流程-1] 配置与客户端就绪，进入登录阶段`。

**869 相关**：`Protocol.version == "869"` 时使用 Client869；后续登录、WS 的 key 均与 869 约定一致（token_key / admin_key）。

### 步骤 2：等待登录（流程-2）

**做什么**：恢复或新建登录态；未登录则拉二维码并轮询扫码结果，直到登录成功。

**代码位置**：`bot_core.py` 约 684–784 行（状态与 robot_stat）、约 1062–1620 行（登录循环与成功分支）。

**具体动作**：

1. **状态与日志**  
   - `update_bot_status("waiting_login", "等待微信登录")`。  
   - 打日志：`[流程-2] 等待登录（扫码或恢复会话）`。

2. **robot_stat.json**  
   - 路径：`script_dir / "resource" / "robot_stat.json"`。  
   - 若不存在则创建，字段含：`wxid`、`token_key`、`poll_key`、`display_uuid`、`login_tx_id`、`device_type` 等。  
   - 若存在则读取，用于 869 恢复会话。

3. **869 恢复会话（仅当 protocol_version == "869" 且 token_key 存在）**  
   - 从 robot_stat 恢复：`bot.token_key`、`bot.poll_key`、`bot.device_type`、`bot.device_id`、`bot.display_uuid` 等。  
   - 若 robot_stat 中有 `wxid`：设 `bot.wxid`，并调 `bot.is_logged_in(wxid)` 验证；通过则视为已登录，跳过扫码；不通过则清空 wxid/token_key 等并进入扫码流程。

4. **未登录时：扫码登录循环**  
   - `update_bot_status("offline", "未登录，准备获取二维码")`。  
   - `while not await bot.is_logged_in(wxid):` 内：  
     - 869：调 `bot.get_login_qr_code(...)`（内部走 GetLoginQrCodeNewDirect），拿到二维码 URL/数据。  
     - 869：可选从 `bot.get_login_context_snapshot(token_key=...)` 取 `poll_key`、`display_uuid`、`login_tx_id`、设备信息等。  
     - `update_bot_status("waiting_login", "等待微信扫码登录", { uuid, display_uuid, poll_key, ... })`。  
     - 轮询：`stat, data = await bot.check_login_uuid(current_poll_key, device_id=...)`（869 侧为 CheckLoginStatus 等）。  
     - 若返回已登录：取 `wxid`，做 Profile 一致性校验 `_ensure_login_profile_consistency(login_wxid=wxid)`；通过则 `break` 跳出循环；不通过则清空 token_key 等并继续扫码。

5. **登录成功**  
   - 从 `data` 或 bot 取 `wxid`，写回 robot_stat（token_key、poll_key、wxid、device_type 等）并落盘。  
   - 打日志：`[流程-3] 登录成功，进入初始化阶段`（见下一步开头）。

**869 相关**：869 使用 GetLoginQrCodeNewDirect 获取二维码，CheckLoginStatus 轮询；key 为 token_key（设备授权码）；恢复会话时用 robot_stat 中的 token_key/wxid 并做 is_logged_in 校验。

### 步骤 6：等待消息（流程-6）

**做什么**：确定 WS 地址与 key、启动消息消费者、根据 message_mode 选择 WebSocket 或 HTTP 轮询，并进入收消息循环。

**代码位置**：`bot_core.py` 约 1825–1936 行（WS/HTTP 选择与调用）、约 1942–2350+ 行（`listen_ws_messages`）、约 371–423 行（`message_consumer`）。

**具体动作**：

1. **WebSocket URL 与 869 key**  
   - 从配置读 `ws-url`（或 `ws_url` 等）；若无则用 `host` + `port` 拼 `ws://{host}:{port}/ws/GetSyncMsg`。  
   - 869：不在 URL 后拼 wxid；若 URL 未包含 `key=`，则用 `_resolve_active_sync_key(bot)` 取当前 token_key，或回退到 `admin-key`，再 `_append_sync_key_to_url(ws_url, key)` 得到最终 `ws_url`。  
   - 打日志：`WebSocket 消息推送地址: {ws_url}`。

2. **Redis 与队列**  
   - `redis = aioredis.from_url(redis_url, decode_responses=True)`。  
   - `queue_name = config.get("redis-queue-name", "xbot")`。

3. **启动消息消费者**  
   - `for _ in range(NUM_CONSUMERS): asyncio.create_task(message_consumer(xybot, redis, message_db, queue_name))`。  
   - `message_consumer` 内：`redis.blpop(queue_name)` 阻塞取消息 → `_validate_inbound_message(message, source="consumer")` → 通过则 `await xybot.process_message(message)`，否则丢弃并打日志。

4. **消息模式分支**  
   - `message_mode = config 或 app_config 的 message-mode / message_mode`，默认 `"ws"`。  
   - **若 message_mode == "http"**：`await http_poll_messages(xybot, api_host, api_port, wxid, redis, message_db, queue_name, api_prefix=...)`，轮询 869 HttpSyncMsg 等。  
   - **否则（ws）**：`await listen_ws_messages(xybot, ws_url, redis, message_db, queue_name, http_fallback={...}, ws_fail_threshold=...)`。

5. **listen_ws_messages（869 收消息核心）**  
   - `websockets.connect(runtime_ws_url, ...)` 连接 `ws://.../ws/GetSyncMsg?key=...`。  
   - 循环 `msg = await websocket.recv()` → `data = json.loads(msg)`。  
   - **取消息数组**：优先 `data.AddMsgs`/`addMsgs`/`messages` 等；若无则看 `data.Data`/`data.payload` 内是否有上述列表；**869 特有**：若 `data.message` 为 dict 且无子列表，则 `messages = [data["message"]]`。  
   - 对每条 `message`：先 `_validate_inbound_message(message, source="ws_addmsgs")`；若不通过则做 **869 适配**（从 `from_user_name`、`msg_id`、`content` 等提取，构造标准 addmsg_869），再校验；通过则落库、去重、`redis.rpush(queue_name, json.dumps(message))`。  
   - 若 `messages is None`（非 AddMsgs 包）：走 else 分支，对根包 `message` 解包后同样做发送者/内容/ID 提取，构造 addmsg，校验后入队。  
   - WS 连续失败达阈值时，可降级到 `http_poll_messages`（由 `http_fallback` 提供参数）。

6. **结果**  
   - 入队消息为标准 AddMsgs 格式（含 `FromUserName`、`MsgId`、`Content` 等）；消费者用 `process_message` 交给插件，回复时 `ToUserName` 为解析出的真实发送者 wxid，从而完成「从登录到等待接收消息」的整链。

**869 相关**：WS 地址为 `/ws/GetSyncMsg`，key 为 token_key（或回退 admin_key）；推送根包为 `{ key, message, type }`，内层为 snake_case（如 `from_user_name`、`msg_id`）；适配逻辑见本文档第 5–8 节。

以上 6 步串联起来即：**配置与客户端就绪 → 等待登录（恢复/扫码）→ 登录成功与 Profile → 初始化（心跳、XYBot、DB、插件）→ 就绪 → 等待消息（WS 或 HTTP）**。869 在步骤 2、3、6 中与协议约定对齐（key、接口、推送格式、收消息适配）。

## 1. 适配目标与问题背景

### 1.1 目标

- **收消息**：框架通过 `ws://host:port/ws/GetSyncMsg?key=<token_key>` 接收 869 推送，从「初始化 → 登录 → 能力就绪 → 等待消息」整条流程跑通。
- **解析**：能从 869 推送中正确解析 **发送者（sender/wxid）**、**消息内容**、**消息 ID**、**消息类型**，满足门禁与下游插件要求。
- **发消息**：回复时使用解析出的真实发送者 wxid 作为 `ToUserName`，确保消息能发到对方而非占位符。

### 1.2 遇到的问题

1. **消息结构不一致**：869 WS 推送根对象为 `{ key, message, type }`，真实消息在 `message`（dict）内；内层字段多为 **snake_case**（如 `from_user_name`、`msg_id`、`msg_type`），与框架预期的 **PascalCase / 混合**（如 `FromUserName`、`MsgId`）不一致。
2. **门禁丢弃**：入站消息经 `_validate_inbound_message` 校验，依赖 `_collect_message_core_fields` 提取 sender、msg_type、message_id。869 内层未提供 `FromUserName`/`MsgId` 等键时，门禁报 `missing_fields=['msg_type', 'sender', 'message_id']`，整条被丢弃。
3. **发送者占位符**：若发送者未解析，框架使用 `unknown_869` 作为 sender，回复发往 `ToUserName: unknown_869`，协议端无法投递，表现为「收得到消息但发不出去」。

## 2. 869 WS 推送格式（协议侧）

### 2.1 根包结构

| 键 | 说明 |
|----|------|
| `key` | 会话/设备 key（与 URL 的 `?key=` 一致） |
| `message` | 单条消息对象（dict）或内嵌列表的容器 |
| `type` | 类型标识（可选） |

- 当 `message` 为 **dict** 且无 `AddMsgs`/`addMsgs`/`messages` 等列表键时，框架将该 dict 视为 **单条消息**，即 `messages = [data["message"]]`，进入「有消息列表」分支处理。

### 2.2 内层单条消息格式（实测）

869 协议侧单条消息常见键名（snake_case）如下：

| 协议键名 | 含义 | 框架侧对应 |
|----------|------|-------------|
| `msg_id` | 消息 ID | `MsgId` / `message_id` |
| `new_msg_id` | 新消息 ID | `NewMsgId` |
| `from_user_name` | **发送者 wxid** | `FromUserName.string` |
| `to_user_name` | 接收者 wxid | `ToUserName.string` |
| `msg_type` | 消息类型 | `MsgType` |
| `content` | 文本内容 | `Content.string` |
| `status` | 状态 | - |
| `img_status` / `img_buf` | 图片相关 | - |
| `create_time` | 创建时间 | - |
| `msg_source` | 消息来源 XML | `MsgSource` |
| `push_content` | 推送摘要 | `PushContent` |
| `msg_seq` | 消息序号 | `MsgSeq` |

框架侧统一使用的 **标准 AddMsgs 条目** 为 PascalCase + 嵌套 `{ "string": "..." }` 的格式，见下文「框架标准格式」。

## 3. 框架侧标准格式（门禁与下游约定）

门禁与下游（xybot、插件）期望的单条消息结构示例：

```json
{
  "MsgId": 1234567890,
  "NewMsgId": 1234567890,
  "MsgSeq": 0,
  "FromUserName": { "string": "wxid_xxx" },
  "ToUserName": { "string": "wxid_bot" },
  "MsgType": 1,
  "Content": { "string": "消息内容" }
}
```

- **发送者**：`FromUserName.string` 或由 `_collect_message_core_fields` 解析出的 `sender`，下游会转为 `FromWxid` / `SenderWxid`。
- **回复目标**：发送接口使用 `ToUserName` = 发送者 wxid（即 `FromUserName.string`）。
- **消息 ID**：`MsgId` / `NewMsgId` 用于去重、落库与日志。

## 4. 代码改动总览

### 4.1 涉及文件

| 文件 | 改动概要 |
|------|----------|
| `bot_core.py` | WS 消息解析、消息数组来源、AddMsgs 分支 869 适配、门禁字段提取、else 分支 869 解包与适配 |

### 4.2 涉及函数与逻辑块

- `_extract_message_text(value)`：兼容 dict/string，用于从嵌套对象中取文本（如 `{"string":"wxid_xxx"}`）。

- `listen_ws_messages` 内：
  - **消息数组来源**：支持根包 `data.message` 为单条 dict 时 `messages = [inner]`。
  - **AddMsgs 分支**：当门禁不通过时，对当前条做 **869 格式适配**，再校验、入队。
  - **else 分支**：对根包 `message` 解包，兼容 869 多种发送者/内容键名，构造标准 addmsg 后入队。

## 5. 消息数组来源（统一取 messages）

**位置**：`listen_ws_messages` 内，`data = json.loads(msg)` 之后。

**逻辑**（按优先级）：

1. 根包 `data` 的 `AddMsgs` / `addMsgs` / `messages` / `Messages` / `list` / `List` 中任一为 list → 取为 `messages`。
2. 根包 `data.Data` / `data.payload` 等为 dict 时，再从其内取上述 list 键。
3. **869 特有**：根包 `data.message`（或 `data.Message`）为 dict 时：
   - 若其内仍有 `AddMsgs`/`addMsgs`/… 列表 → 取该列表为 `messages`；
   - 否则将 **该 dict 视为单条消息**：`messages = [data["message"]]`。

这样 869 的 `{ key, message, type }` 且 `message` 为单条对象时，会进入「有 messages 列表」分支，逐条做门禁与适配。

## 6. 门禁与核心字段提取（_collect_message_core_fields）

**作用**：从任意一条「可能是 869 或框架标准」的消息对象中，统一抽出 `msg_type`、`msg_id`、`new_msg_id`、`message_id`、`sender`、`to_wxid`、`content`，供门禁校验与后续落库/入队使用。

### 6.1 发送者（sender）提取顺序

在已有 FromUserName / FromWxid / sender / from_user / FromUser / from / fromUserName / Talker 等基础上，**新增 869 常见键**：

1. `FromUserName`（含嵌套 `string`/`str` 等，由 `_extract_message_text` 处理）
2. `FromWxid`
3. `sender`（对象时取 id/string/UserName 等）
4. `from_user`、`FromUser`、`from`（字符串或对象）
5. `fromUserName`
6. **`from_user_name`**（869 snake_case 发送者，本次新增）
7. `Talker`、`talker`（字符串或对象）

### 6.2 内容（content）提取

已兼容：`Content`、`content`、`TextContent`、`textContent`、`Text`、`text`、`MsgContent`、`msgContent`（含嵌套由 `_extract_message_text` 处理）。

### 6.3 消息类型与 ID

- **msg_type**：`MsgType` → `msgType` → `category`（未增加 snake_case `msg_type`，避免 869 原始包直接通过门禁被原样入队，下游依赖标准键名）。
- **msg_id**：`MsgId` → `msgId`。
- **new_msg_id**：`NewMsgId` → `newMsgId`。

门禁要求：`sender`、`msg_type`、`message_id` 均有效才通过。869 内层多为 `msg_type`/`msg_id` 等 snake_case，不在此处映射，从而保证 869 条目不直接通过门禁，而是走「适配 → 再校验」路径，入队的是标准 AddMsgs 结构。

---

## 7. AddMsgs 分支中的 869 适配（核心）

**位置**：`listen_ws_messages` 中，`for message in messages` 循环内，当 `_validate_inbound_message(message, source="ws_addmsgs")` 返回 `valid=False` 时。

**思路**：将当前 `message` 视为 869 内层格式，用与 else 分支一致的多种键名提取发送者、内容、ID、类型，**构造框架标准 addmsg**，再对构造结果做一次门禁校验；通过则用该标准对象作为本条消息入队并落库。

### 7.1 原始字段读取（兼容多种命名）

- **消息 ID**  
  `msg_id`：`msgId` / `MsgId` / `id` / `message_id` / **`msg_id`**  
  `new_msg_id`：`newMsgId` / `NewMsgId` / **`new_msg_id`** / 上述 msg_id  
  `msg_seq`：`msgSeq` / `MsgSeq` / **`msg_seq`**

- **发送者**（按顺序，直至得到非空字符串）  
  1. `sender` / `Sender` 对象 → `id` / `Id` / `string` / `UserName` / `userName`  
  2. `FromUserName` / `fromUserName` / **`from_user_name`**（字符串或 dict，dict 用 `_extract_message_text`）  
  3. `from_user` / `FromUser` / `from`（字符串或 dict）  
  4. `FromWxid` / `SenderId` / `Talker` / `talker`  
  5. `Talker` 为 dict 时用 `_extract_message_text`  
  6. 若仍为空：打 DEBUG 日志「未能解析发送者，原始键: list(msg.keys())」，再尝试键：`From`、`User`、`SenderWxid`、`SenderUserName`、`senderUserName`、`FromWxid`  
  7. 再尝试 **根包 data**：`data.from_user` / `data.FromUser` / `data.sender`（兼容 WSG 等把发送者放在根包的情况）  
  8. 若仍为空：置为 `unknown_869`

- **内容**  
  `content` / `Content` / `TextContent` / `textContent` / `Text` / `text` / `MsgContent` / `msgContent`，同上用 `_extract_message_text` 或 str。

- **消息类型**  
  `category` / `MsgType` / `msgType`，用 `_safe_int(..., 1)`。

### 7.2 构造标准 addmsg_869

```python
addmsg_869 = {
    "MsgId": _safe_int(_raw_msg_id, 0) 或 0,
    "FromUserName": {"string": _sender_str or ""},
    "ToUserName": {"string": 当前 bot 的 wxid},
    "MsgType": _msg_type,
    "Content": {"string": _content_str or ""},
    "NewMsgId": _safe_int(_raw_new_msg_id, 0) 或 0,
    "MsgSeq": _safe_int(_raw_msg_seq, 0) 或 0,
}
```

- 若 `MsgId` 与 `NewMsgId` 均为 0 但已有发送者或内容，则用 **fallback**：  
  `abs(int(time.time()*1000) + hash(content|sender) % 100000)` 作为 MsgId/NewMsgId，避免因缺 ID 被门禁丢弃。

### 7.3 后续流程

- 对 `addmsg_869` 调用 `_validate_inbound_message(..., source="ws_addmsgs")`。
- 通过则：`message = addmsg_869`，继续执行 **去重 → 落库 → 标记已入队 → 入队**，入队内容为 `json.dumps(message, ensure_ascii=False)`（即标准格式）。
- 不通过则：打 WARNING，`dropped_invalid_messages += 1`，`continue`。

---

## 8. else 分支（非 AddMsgs 的 WS 包）

**位置**：`messages is None` 时，对 `data` 或 `data` 拆成的列表逐条处理。

- **869 解包**：若当前条有 `message` / `Message` 且为 dict，则用该 dict 作为后续处理的 `msg`（内层单条）。
- **发送者/内容/ID 提取**：与 AddMsgs 分支类似，兼容 FromUserName、fromUserName、sender、from_user、FromUser、FromWxid、Talker 等；**未在此分支单独加 from_user_name**，因 869 当前走的是「data.message 为 dict → messages = [inner]」的 AddMsgs 分支。
- **构造 addmsg**：与上节结构一致，含 MsgId、FromUserName、ToUserName、MsgType、Content、NewMsgId、MsgSeq 等，无 MsgId 时同样使用 fallback。
- **校验与入队**：对 addmsg 做门禁，通过则落库并 `rpush` addmsg（标准格式）。

## 9. 辅助工具

### 9.1 _extract_message_text(value)

- 若 `value` 为 str，直接返回。
- 若为 dict，依次取键 `string`、`str`、`String`、`value`、`Value`、`id`、`wxid`、`UserName`、`userName`，第一个非空即返回其字符串形式。
- 否则返回 `""`。  
用于统一处理 869 与框架中「可能是字符串或 { string/str/id } 对象」的字段。

### 9.2 _safe_int(value, default=0)

- 空值或非法转换时返回 `default`，避免 None 或非数字导致异常。

## 10. 日志与排查

- **未能解析发送者时**（AddMsgs 分支）：  
  `[WS_869_ADDMSGS] 未能解析发送者，原始键(便于对接): ['msg_id', 'from_user_name', ...]`  
  便于确认协议实际下发的键名，便于继续扩展映射。
- **门禁丢弃**：  
  `[WS_INBOUND_GUARD] source=ws_addmsgs missing_fields=... discard=...`  
  表示该校验未通过（通常为未走适配或适配后仍缺字段）。
- **入队成功**：  
  `消息已入队到队列 xbot，消息ID: xxx`  
  表示本条已为标准格式并写入 Redis，供消费者与插件使用。

## 13. 小结

- **协议格式**：869 WS 根包为 `{ key, message, type }`，内层单条为 snake_case（如 `from_user_name`、`msg_id`、`msg_type`、`content`）。
- **框架格式**：门禁与下游使用 PascalCase + `FromUserName/Content` 等嵌套 `{ "string": "..." }` 的标准 AddMsgs 条目。
- **适配方式**：在 AddMsgs 分支对门禁未通过的条目，按 869 多种键名提取发送者、内容、ID、类型，构造标准 addmsg 后再次门禁；通过则入队与落库均为标准格式，回复使用真实 sender，消息可正常发出。
- **扩展**：若协议新增键名，可在「发送者/内容/ID」提取顺序中增加对应键，或在未能解析时根据 DEBUG 日志中的「原始键」补充映射。
